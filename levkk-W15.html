pgdogApr 8
FEATURE

pgdog now generates 53-bit unique IDs and database-backed sharded sequences

*Distributed sequences no longer require app-side coordination or external services.*

@levkk's #858 adds a native unique ID generator and sharded sequence support directly in pgdog. Before this, applications either generated IDs client-side (losing sharding information) or relied on external services. Now sequences can be defined per shard and pulled through the connection pool, letting pgdog route each nextval() to the right backend and maintain proper ordering within a shard. The 53-bit ID space keeps them JavaScript-safe while staying dense enough for high-cardinality use cases.

pgdogApr 8
FEATURE

dropping child table rows during streaming replication no longer stalls resharding

*Cascading deletes on sharded child tables now propagate correctly through replication.*

@levkk's #880 fixes a bug where deleting rows from a sharded parent table—which cascades to child tables—would fail to replicate those child row deletions during resharding. The replication logic wasn't handling the cascade properly, causing rows to linger on the target shard and diverging the data. Now cascade deletes are recognized and applied in the correct order during the copy-to-replication handoff.

pgdogApr 8
FEATURE

SELECT FOR UPDATE inside CTEs now routes to write replicas correctly

*Applications wrapping row locks in WITH clauses no longer get routed to read-only backends.*

pgdog's query classification logic was missing SELECT FOR UPDATE inside Common Table Expressions (CTEs), routing them to read-only replicas and triggering PostgreSQL's "cannot execute SELECT FOR UPDATE in a read-only transaction" error. A user reported this when their application wrapped a FOR UPDATE check inside a CTE (see issue #878); @levkk's #879 fixes detection by making it recursive—now the parser walks into CTEs and spots the locking clause. The query now correctly routes to the write replica.

pgdogApr 7
FEATURE

table LSN tracking now survives the COPY-to-replication transition

*Resharding no longer loses position during the handoff from bulk copy to logical replication.*

When resharding switched from COPY (bulk data transfer) to logical replication (streaming new changes), pgdog was losing the table's Log Sequence Number (LSN)—the write-ahead log position needed to know where to resume. Without it, subsequent changes could be skipped or replayed incorrectly. @levkk's #874 preserves the LSN across the handoff, ensuring no data is missed or duplicated when replication takes over.

pgdogApr 7
FEATURE

omnisharded resharding copy and replication now coordinate correctly

*Multi-shard resharding no longer stalls waiting for the wrong backend to finish.*

@levkk's #869 fixes a race condition in omnisharded resharding where the COPY phase and replication phase weren't coordinating on which backend had actually finished. This caused resharding to wait indefinitely or proceed prematurely, corrupting the shard split. The fix tightens the synchronization between copy completion and replication slot catchup.

pgdogApr 7
FEATURE

resharding now warns when rows disappear during replication

*Data loss during resharding is no longer silent.*

@levkk's #871 adds logging when resharding detects that row counts don't match between the COPY phase and the final replicated state. If a table has 1000 rows after bulk copy but only 950 rows after replication, the warning surfaces immediately rather than leaving operators guessing why their data shrank. This is especially valuable in omnisharded setups where data flows across multiple shards and tracking consistency is harder.

pgdogApr 6
FEATURE

query cache default limit raised to 1000 entries

*pgdog now caches 10x more queries by default, reducing parser load.*

@levkk's #870 bumps the default query_cache_limit from 100 to 1000. Most production pgdog deployments see more than 100 unique query patterns, so the cache was evicting frequently-used queries. With a higher default, applications avoid re-parsing the same queries on every request—a small but consistent win on CPU and latency, especially in high-volume proxy workloads.

RELEASE

pgdog enterprise control plane Helm chart released

*Operators can now deploy the pgdog control plane on Kubernetes with a single chart.*

@levkk released v0.1.10 of helm-ee, the Helm chart for pgdog's enterprise control plane. This is the initial release (#1), allowing Kubernetes deployments to manage pgdog's configuration, monitoring, and orchestration without hand-rolling manifests. Teams running pgdog on Kubernetes can now integrate it into their standard Helm-based deployment workflows.
